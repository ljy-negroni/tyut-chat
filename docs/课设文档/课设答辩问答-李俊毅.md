## 一、项目整体

### Q1：请用 2 分钟介绍你的项目。

**答：** 我负责开发的是 TYUT CHAT，一个面向太原理工大学师生的校园即时通讯 Web 应用，对应课设题目 3.2"校园即时通信与文件传输系统"。

**架构方面**，我设计了四模块后端 + 独立前端的架构。后端分为公共包（chat-common）、推送 SDK（chat-client）、WebSocket 推送服务（chat-server，基于 Netty，端口 8878）和业务平台（chat-platform，基于 Spring Boot 3.3，端口 8088）。前端使用 Vue 2.7 + Element UI。

**核心功能**包括用户注册登录（JWT 双 Token 认证 + 图形验证码）、私聊与群聊消息（WebSocket 实时推送）、图片/文件/语音发送（MinIO 对象存储）、好友管理、消息已读未读状态、消息撤回删除、WebRTC 音视频通话信令、敏感词过滤。

**技术亮点**有三个：第一，基于 Redis List 实现了多 chat-server 节点的跨节点消息路由，每个节点独立消费自己的队列；第二，JWT accessToken + refreshToken 双 Token 机制实现无状态认证与自动续期；第三，整套 UI 采用 macOS Vibrancy 暗色设计语言，Three.js 3D 粒子背景。

### Q2：项目的四模块架构是如何设计的？模块之间的依赖关系是怎样的？

**答：** 我设计的四模块架构遵循**严格的单向依赖**原则：

```
chat-common（公共包）
    ↑
chat-client（推送 SDK，仅依赖 common）
    ↑
chat-platform（业务平台，依赖 client + common）

chat-common
    ↑
chat-server（推送服务，仅依赖 common）
```

设计约束：
- `chat-common` 定义共享模型（IMTerminalType 枚举）、JWT 工具类（JwtUtil）、通用常量，不依赖任何兄弟模块
- `chat-client` 封装 Redis List 的消息入队操作，作为 SDK 供 platform 调用，屏蔽消息队列细节
- `chat-platform` 是业务核心，处理用户、好友、群组、消息、文件等 HTTP 接口，需要推送消息时调用 chat-client
- `chat-server` 是独立推送进程，消费 Redis 队列，通过 WebSocket 向客户端推送消息

这样设计的目的是**解耦推送与业务**。platform 可以水平扩展处理 HTTP 请求，server 可以水平扩展支撑 WebSocket 连接，两者通过 Redis 队列异步通信，互不阻塞。

### Q3：为什么选择 Spring Boot 3.3.1 + Netty 这个技术栈？

**答：** 技术选型时我对比了多种方案：

| 对比维度 | Spring Boot 3.3 + Netty | Spring WebSocket | Node.js + Socket.io |
|----------|------------------------|-------------------|---------------------|
| HTTP 业务开发效率 | ⭐⭐⭐⭐⭐（Spring 生态） | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
| WebSocket 并发性能 | ⭐⭐⭐⭐⭐（NIO） | ⭐⭐⭐（BIO，每连接一线程） | ⭐⭐⭐⭐ |
| 数据库操作 | ⭐⭐⭐⭐⭐（MyBatis-Plus） | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
| 学习曲线 | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |

选择 Spring Boot 3.3 + Netty 的理由：
1. **业务与推送分离**：Spring Boot 擅长 REST API + 数据库操作，Netty 擅长高并发长连接
2. **Netty 的 NIO 模型**：基于 EventLoop 线程池 + Selector 多路复用，单机可支撑万级 WebSocket 连接，内存占用远低于 BIO 模型
3. **Spring 生态集成**：MyBatis-Plus 简化 CRUD、Druid 连接池监控、springdoc-openapi 自动生成 API 文档（兼容 Swagger 3 规范）、Spring Validation 参数校验
4. **JDK 17 特性**：record 类、密封类、Pattern Matching 等新特性提升代码可读性

## 二、数据库设计

### Q5：数据库表结构是怎样设计的？画一下 ER 图。

**答：** 核心 8 张表及其关系：

```
im_user（用户表）
  ├── 1:N → im_friend（好友关系表，user_id / friend_id 均外键引用 im_user.id）
  ├── 1:N → im_group_member（群成员表）
  ├── 1:N → im_private_message（私聊消息表，send_id / recv_id）
  └── 1:N → im_group_message（群聊消息表，send_id）

im_group（群组表）
  ├── 1:N → im_group_member（群成员表，group_id 外键）
  └── 1:N → im_group_message（群聊消息表，group_id 外键）

im_sensitive_word（敏感词表）—— 独立字典表
im_message_deletion（消息删除记录）—— 关联消息 ID
- im_file_info（文件信息表）—— 记录上传文件元数据
```

关键字段设计：
- `im_user`：password 使用 BCrypt（$2a$10$）加密存储，不存明文；head_image 与 head_image_thumb 分别存原图与缩略图 URL
- `im_private_message`：type 字段区分消息类型（0=文字、1=图片、2=文件、3=语音、4=视频），status 字段追踪消息状态（0=未送达 PENDING、1=已送达 DELIVERED、2=已撤回 RECALL、3=已读 READED）。（前端额外维护本地状态：-2=发送失败、-1=发送中），content 字段存 JSON 格式（文字直接存字符串，图片存 `{"originUrl":"...", "thumbUrl":"...", "width":1280, "height":1792}`）
- `im_friend`：version 字段用于增量同步好友列表变更，避免全量拉取
- 所有表使用 InnoDB 引擎（支持事务）、utf8mb4 字符集（支持 emoji）、自增主键 BIGINT

### Q6：为什么要将消息内容存为 JSON 格式而不是多个字段？

**答：** 采用 JSON 格式存储消息内容是一种**多态消息体**的设计模式。

不同类型的消息有不同的数据结构：
- 文字消息：`"你好"`（纯字符串）
- 图片消息：`{"originUrl":"...","thumbUrl":"...","width":1280,"height":1792}`
- 文件消息：`{"name":"报告.pdf","url":"...","size":2048000}`
- 语音消息：`{"url":"...","duration":15}`

如果拆成多个字段，图片消息的 width/height 字段对文字消息就是浪费，而且每增加一种消息类型就要 ALTER TABLE 加字段。JSON 格式的优点：
1. **扩展性好**：新增消息类型只需前端约定 JSON schema，后端不改表结构
2. **查询友好**：MySQL 5.7+ 支持 JSON_EXTRACT 函数，可针对 JSON 内字段建虚拟列索引
3. **前后端统一**：前端发送的 JSON 直接存入数据库，读取时直接返回给另一个客户端，无需转换

### Q7：索引是怎么设计的？为什么要这样建索引？

**答：** 索引设计遵循"以查询驱动"原则：

| 表 | 索引 | 索引类型 | 覆盖查询场景 |
|----|------|---------|-------------|
| im_user | user_name | UNIQUE | 登录验证、注册查重 |
| im_friend | (user_id, friend_id) | 联合索引 | "查我的所有好友"WHERE user_id=? |
| im_private_message | (send_id, recv_id, send_time) | 联合索引 | 私聊历史消息分页 ORDER BY send_time DESC |
| im_private_message | send_time | 单列索引 | 消息时间范围查询 |
| im_group_message | (group_id, send_time) | 联合索引 | 群聊历史消息分页 |
| im_group_member | (group_id, user_id) | 联合索引 | "群内所有成员"+"检查某人是否在群内" |

设计考量：
- **联合索引最左匹配**：`(send_id, recv_id, send_time)` 可同时覆盖"查 A 发给 B 的消息"（send_id + recv_id）和"查 A 和 B 之间的所有消息"（send_id + recv_id 固定，按 send_time 排序）
- **避免过度索引**：单表索引不超过 5 个，写入性能与查询性能平衡
- **覆盖索引**：常用查询的 SELECT 字段尽量包含在索引中，减少回表

## 三、WebSocket 实时通信

### Q8：WebSocket 推送服务的完整工作流程是什么？

**答：** 我以用户 A 向用户 B 发送消息为例，完整链路如下：

```
[用户 A 浏览器]
    │ 1. HTTP POST /message/private/send
    ▼
[chat-platform] 
    │ 2. 消息持久化 MySQL
    │ 3. 调用 IMClient.push(message, recvId)
    ▼
[chat-client SDK]
    │ 4. Redis GET im:user:server_id:{userId}:{terminal}  → 查到 serverId
    │ 5. Redis LPUSH im:unread:{serverId} messageJson
    ▼
[Redis List]
    │ 6. chat-server 线程 BLPOP im:unread:{serverId}
    ▼
[chat-server]
    │ 7. 查找 userId→Channel 本地映射
    │ 8. Channel.writeAndFlush(TextWebSocketFrame)
    ▼
[用户 B 浏览器]
    │ 9. WebSocket onMessage 接收
    │ 10. 发送 ACK 确认帧
    ▼
[chat-server → chat-platform]
    │ 11. 更新消息 status = DELIVERED
```

关键时间节点：步骤 3-8 在局域网内 < 100ms；步骤 1 到 10 完整耗时 < 200ms，符合实时通信要求。

### Q9：如果用户 B 不在线，消息怎么处理？

**答：** 分两种情况：

**情况一：B 离线（未连接 WebSocket）**
1. 消息正常保存到 MySQL（im_private_message 表，status = PENDING）
2. IMClient 查询 Redis `im:user_server:{recvId}` 返回 null（无在线 server）
3. 消息不入队，直接返回（消息已在数据库，等用户上线拉取）

**情况二：B 上线**
1. chat-server 的 LoginProcessor 处理登录帧
2. 更新 Redis：`SET im:user_server:{userId} {serverId}`
3. 从 MySQL 拉取该用户的所有 PENDING 消息：`SELECT * FROM im_private_message WHERE recv_id = ? AND status = 0`
4. 批量推送给用户，更新状态为 DELIVERED

**情况三：B 已读消息**
1. B 打开与 A 的会话时，前端发送 READ 帧
2. chat-platform 批量更新：`UPDATE im_private_message SET status = 3 WHERE send_id = ? AND recv_id = ? AND status < 3`
3. chat-platform 推送已读回执给 A，A 的消息变为"已读"状态

### Q10：Netty 的线程模型是怎样的？为什么能支撑高并发？

**答：** Netty 采用主从 Reactor 多线程模型：

```
BossGroup（1 个 NioEventLoop）
  │ 负责：监听端口、Accept 新连接
  │ 将新 Channel 注册到 WorkerGroup
  ▼
WorkerGroup（CPU 核心数 × 2 个 NioEventLoop）
  │ 负责：读写 I/O、编解码、ChannelHandler 业务处理
  │ 每个 EventLoop 绑定一个 Selector，处理多个 Channel
  ▼
业务线程池（可配置大小）
  │ 负责：耗时操作（数据库查询、Redis 操作）
  │ 避免阻塞 I/O 线程
```

性能分析：
- **NIO 多路复用**：一个线程管理数千个连接，不需要为每个连接创建线程
- **零拷贝**：Netty 的 ByteBuf 支持堆外内存，文件传输时使用 FileRegion 实现零拷贝
- **池化内存**：PooledByteBufAllocator 减少 GC 压力
- **本项目配置**：BossGroup=1, WorkerGroup=4, 业务线程池=CPU核心数×2（动态计算）。本地测试 1000 并发连接 CPU 占用 < 5%

### Q11：WebSocket 连接断开后如何处理？心跳机制是怎样的？

**答：** 三层面保障连接可靠性：

**服务端心跳检测：**
- Netty pipeline 中添加 `IdleStateHandler(60, 0, 0)`：60 秒未收到客户端任何数据 → 触发 `userEventTriggered(IdleState.READER_IDLE)`
- 触发后关闭该 Channel，清理 userId→Channel 映射，删除 Redis 在线状态

**客户端心跳发送：**
- 每 30 秒自动发送 PING 帧（JSON：`{"type": 99}`）
- 超过 60 秒未收到任何帧 → 判定连接断开 → 自动重连

**自动重连机制：**
- 断开后启动重连定时器，使用固定 15 秒延迟自动重连
- 重连成功后重新发送登录帧，恢复 userId→Channel 映射
- 重连期间收到的离线消息在连接恢复后批量推送

## 四、认证与安全

### Q12：JWT 认证的完整流程是什么？Token 如何防止被盗用？

**答：** 采用双 Token 机制：

**Token 生成（登录时）：**
1. 用户提交 userName + password
2. 后端验证密码（BCrypt.matches）
3. 生成 UserSession JSON（userId, terminal, loginTime）
4. 签名生成 accessToken：`JWT.sign(userId, sessionJson, 1800, accessSecret)`
5. 签名生成 refreshToken：`JWT.sign(userId, sessionJson, 7day, refreshSecret)`
6. 两个 Token 使用不同的密钥签名，互不通用

**Token 使用：**
- accessToken 存 sessionStorage，每次请求通过 `accessToken` 请求头携带
- AuthInterceptor 拦截所有请求（白名单：/login /register /captcha /refreshToken /logout /swagger-ui/** /v3/api-docs/** 等），验证签名 + 有效期

**Token 续期（accessToken 过期）：**
1. 前端 axios 响应拦截器捕获 401
2. 用 refreshToken 调用 `PUT /refreshToken`
3. 后端验证 refreshToken → 颁发新 accessToken + 新 refreshToken（旧 refreshToken 仍有效，但建议滚动更新）
4. 用新 accessToken 重放原请求

**防窃取措施：**
- accessToken 有效期短（15 分钟），即使被截获影响时间有限
- refreshToken 仅用于续期接口，使用频率低，暴露面小
- 登录时校验验证码，防止暴力破解获取 Token
- BCrypt 加密存储密码，即使数据库泄露也无法还原明文

### Q13：BCrypt 加密原理是什么？为什么选它？

**答：** BCrypt 是基于 Blowfish 加密算法的密码哈希函数，核心原理：

1. **加盐**：每次加密自动生成 16 字节随机盐值，相同密码两次加密结果不同
2. **多轮哈希**：cost factor（工作因子）控制迭代次数，$2a$10$ 表示 2¹⁰ = 1024 轮
3. **输出格式**：`$2a$10$[22字符盐值][31字符哈希值]`，盐值嵌入结果中，验证时自动提取

选择 BCrypt 而非 MD5/SHA256 的原因：
- **抗暴力破解**：1024 轮迭代使单次验证成本可控（~100ms），而攻击者尝试千万级密码组合耗时剧增
- **内置加盐**：不需要开发者手动管理盐值，避免常见安全漏洞
- **可调节强度**：硬件性能提升后可增加 cost factor（如 $2a$12$）

### Q14：敏感词过滤是怎么实现的？

**答：** 基于数据库 + 内存缓存的敏感词过滤：

1. **数据存储**：敏感词存储在 `im_sensitive_word` 表中（content、enabled），支持动态增删
2. **启动加载**：chat-platform 启动时将所有 enabled=true 的敏感词加载到内存
3. **定时刷新**：每 60 秒执行 ReloadSensitiveWordTask，从数据库重新加载，支持不停机更新
4. **过滤算法**：使用自定义 Trie 前缀树实现（SensitiveFilterUtil），多模式匹配一次扫描即可检测所有敏感词，时间复杂度 O(n)
5. **触发点**：用户注册（用户名 + 昵称）、发送消息（消息文本内容）

## 五、消息系统

### Q15：消息的完整生命周期是怎样的？

**答：** 一条消息从发送到被阅读，经历以下状态流转：

```
SENDING（发送中，status=-1）
  │ 消息已组装，等待 HTTP 返回
  ▼
PENDING（未送达，status=0）
  │ 服务端已持久化，但接收方尚未收到推送
  ▼
DELIVERED（已送达，status=1）
  │ 接收方收到推送并返回 ACK
  ▼
READED（已读，status=3）
  │ 接收方打开会话，批量标记已读
  
特殊状态：
RECALL（已撤回，status=2）
  │ 发送者 2 分钟内撤回消息
```

撤回限制：消息发送后 2 分钟内可撤回，超过 2 分钟禁止撤回（后端校验 `sendTime + 2min > now`）

### Q16：消息如何进行分页查询？如何保证不丢消息？

**答：** 分页查询采用**基于 sendTime 的游标分页**而非 offset 分页：

```sql
-- 拉取某会话最新 20 条消息
SELECT * FROM im_private_message
WHERE ((send_id = ? AND recv_id = ?) OR (send_id = ? AND recv_id = ?))
ORDER BY send_time DESC LIMIT 20;

-- 加载更早的消息（使用上次结果中最小的 sendTime 作为游标）
SELECT * FROM im_private_message
WHERE ((send_id = ? AND recv_id = ?) OR (send_id = ? AND recv_id = ?))
AND send_time < ?
ORDER BY send_time DESC LIMIT 20;
```

为什么不用 offset？
- offset 在并发写入时可能重复或遗漏（新消息插入导致偏移）
- 游标分页以 sendTime 为锚点，天然幂等，不会漏也不会重

保证不丢消息：
- 消息先写入 MySQL（持久化），再入 Redis 队列（推送）
- 即使推送失败，用户下次上线时拉取离线消息（WHERE status = PENDING）
- 每条消息有全局唯一的 Snowflake ID（有序、分布式唯一）

### Q17：群聊消息的推送与私聊有什么区别？@提醒怎么实现？

**答：** 群聊消息推送流程：
1. platform 收到群消息 → 写入 MySQL（im_group_message） → 查询群成员列表
2. 遍历群成员，排除发送者自己，对每个在线成员查询其 serverId
3. 将消息 JSON 写入对应 server 的 Redis 队列
4. 各 chat-server 消费队列并推送给目标用户

与私聊的区别：
- 私聊：推送目标固定 1 人
- 群聊：需要群成员查询 + 批量写入多个 Redis 队列（同一 server 的消息合并为一次 LPUSH）

@提醒实现：
- 前端输入框中输入 `@` 触发群成员选择器
- 发送消息时 content 中包含 `@userId` 标记
- platform 解析消息内容，若包含 `@all`（@全体成员）或 `@指定用户`，设置消息的 atMessage 标志
- 接收方会话列表中显示"有人@我"或"@全体成员"红色提示

### Q18：MinIO 是如何集成的？文件上传流程是怎样的？

**答：** MinIO 集成配置：

```yaml
minio:
  endpoint: http://127.0.0.1:9000      # 内网地址
  domain: http://${MINIO_DOMAIN}:9000  # 外网访问地址（环境变量）
  accessKey: admin
  secretKey: admin123456
  bucketName: tyut-chat
  expireIn: 180                         # 文件过期天数
  imagePath: image                      # 图片子目录
  filePath: file                        # 文件子目录
  videoPath: video                      # 视频子目录
  expireIn: 180                         # 文件过期天数
```

文件上传流程：
1. 前端选择文件 → FileUpload 组件校验（图片 ≤5MB / 文件 ≤10MB）
2. HTTP POST `/image/upload` 或 `/file/upload`
3. 后端 FileServiceImpl：
   - 生成 UUID 文件名（避免原始文件名冲突）
   - 图片：Thumbnailator 生成 300px 宽缩略图，上传原图 + 缩略图
   - `minioClient.putObject(bucket, path/uuid.jpg, inputStream, contentType)`
   - 返回原图 URL + 缩略图 URL
4. 前端将 URL 封装到消息 JSON，发送给接收方
5. 接收方加载缩略图，点击可查看原图

确保 bucket 公开可读：
- `FileServiceImpl.init()` 每次启动调用 `setBucketPublic()`
- Policy 允许匿名 GetObject：`{"Version":"2012-10-17","Statement":[{"Effect":"Allow","Principal":{"AWS":["*"]},"Action":["s3:GetObject"],"Resource":["arn:aws:s3:::tyut-chat/*"]}]}`

## 七、项目总结

### Q19：项目中你遇到的最大技术难点是什么？怎么解决的？

**答：** 最大的难点是**WebSocket 多节点推送的跨节点路由问题**。

问题场景：假设有 3 台 chat-server 实例（A、B、C），用户 1 连接在 A 上，用户 2 连接在 B 上。当用户 1 给用户 2 发消息时，chat-platform 如何知道将消息推给 B 而不是 A 或 C？

解决过程：
1. **第一版方案（失败）**：chat-platform 直接持有所有 WebSocket 连接 → 但这违背了"推送与业务分离"的设计初衷，platform 变成有状态服务，无法水平扩展
2. **第二版方案（采纳）**：引入 Redis 作为协调中心
   - 用户上线时，server 记录 `SET im:user_server:{userId} {serverId}`
   - chat-platform 发送消息前，查询 `GET im:user_server:{recvId}` 获取目标 serverId
   - 消息写入对应 server 的队列 `LPUSH im:unread:{serverId} messageJson`
   - 各 server 通过 LPOP 轮询自己的队列
3. **优势**：platform 与 server 完全解耦，通过 Redis 间接通信；server 节点增减对 platform 透明

### Q20：如果重新做这个项目，你会做哪些改进？

**答：** 
1. **引入消息队列中间件**：当前用 Redis List 做消息队列，功能足够但可靠性不如专业的 RabbitMQ/Kafka（持久化、死信队列、消息回溯）
2. **API 网关**：引入 Spring Cloud Gateway 统一入口，集中处理限流、熔断、日志
3. **容器化部署**：编写 Dockerfile + docker-compose.yml，一键启动全部服务（MySQL + Redis + MinIO + chat-server + chat-platform + chat-web）
4. **消息端到端加密**：引入 Double Ratchet 算法，实现真正的端到端加密
5. **自动化测试**：后端增加单元测试（JUnit + Mockito）和集成测试（Testcontainers），前端增加 Jest 组件测试
