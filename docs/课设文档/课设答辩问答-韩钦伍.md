## 一、需求分析

### Q1：你是如何进行需求分析的？用了什么方法？

**答：** 我按照软件工程标准的需求分析流程，分四个阶段进行：

**第一阶段：课设题目解读**
- 仔细研读课设题目 3.2"校园即时通信与文件传输系统"，提取 11 条基本需求
- 逐条分析每项需求的技术含义与验收标准
- 例如"服务器维护在线用户列表"→ 需求分解为：① 用户上线时服务端感知 ② 存储在统一的数据结构中 ③ 用户下线/异常断开时及时移除 ④ 客户端可查询

**第二阶段：业务场景建模**
- 从用户视角出发，梳理典型使用场景：
  - 新生入学注册账号 → 添加同班同学为好友 → 创建班级群聊 → 课间发消息讨论作业 → 发送课件文件 → 期末查看历史消息复习
- 用这个场景串起所有功能需求，确保没有断裂的体验链路

**第三阶段：竞品参考分析**
- 分析了微信、QQ、Discord 三款即时通讯产品
- 提取共性功能（消息气泡、在线状态、未读徽标、文件传输）作为必须实现项
- 提取差异化亮点（Discord 的服务器/频道模型、微信的撤回编辑）作为扩展参考

**第四阶段：需求优先级排序**
- 使用 MoSCoW 方法：
  - Must Have（必须有）：登录注册、私聊文本、群聊、文件传输、在线用户列表、历史消息
  - Should Have（应该有）：消息已读/未读、离线消息、好友管理、消息撤回
  - Could Have（可以有）：音视频通话、文件断点续传、敏感词过滤、管理员广播
  - Won't Have（不做）：移动端 App、朋友圈动态、支付红包、第三方登录

最终产出《需求分析文档》，包含功能需求矩阵、非功能需求、数据流图、用例描述。

### Q2：课设要求中有哪些需求是容易遗漏的？你是如何确保全覆盖的？

**答：** 容易遗漏的需求主要有三类：

**1. 异常场景需求：**
- 课设要求第 10 条："用户退出或异常断开，服务器更新在线状态且不崩溃"
- 这不仅是正常退出，还包括：网络断开、浏览器崩溃、电脑休眠恢复
- 我编写了 5 个异常测试用例：强制关闭浏览器、断开 WiFi、浏览器标签页休眠、同时关闭多个标签页、服务端重启后客户端自动重连

**2. 非功能性需求：**
- 课设题目未明确要求但隐含的：密码安全存储、并发消息不丢失不乱序、大文件传输不阻塞其他消息
- 我在需求文档中补充了安全需求（BCrypt 加密、JWT 鉴权）、性能需求（消息延迟 <500ms）、可靠性需求（消息持久化）

**3. 用户体验细节：**
- 登录页验证码刷新、自动登录、错误提示
- 消息发送状态反馈（发送中 → 已送达 → 已读）
- 这些虽不是课设硬性要求，但直接影响演示效果

确保全覆盖的方法：
- 逐条对照课设题目，在需求矩阵中标记每条需求的实现状态
- 对于每条需求，编写至少 2 个验收用例（正常流程 + 异常流程）
- 用"需求追溯矩阵"确保每条课设要求都有对应的实现和测试

### Q3：为什么选择这个项目定位？与微信/QQ有什么差异化？

**答：** 我们的项目定位是**面向太原理工大学师生的校园即时通讯平台**，与微信/QQ 的差异化体现在：

**1. 校园身份认证**
- 用户名即为校园身份标识，天然与学号/工号关联
- 无需手机号绑定，降低注册门槛
- 微信/QQ 是通用社交工具，缺乏校园专属身份体系

**2. 轻量化与可控性**
- 功能聚焦于通讯本身，无朋友圈、小程序、支付等庞大生态
- 代码量适中（约 15000 行），适合课程设计规模
- 数据部署在校内服务器，学校可自主管理

**3. 定制化扩展空间**
- 后续可接入校园系统（教务通知、图书馆提醒、一卡通消费记录）
- 可扩展为班级管理工具（作业提交、通知公告、投票问卷）
- 这些是微信无法提供的校园垂直场景

**4. 技术学习价值**
- 涵盖 WebSocket 长连接、JWT 认证、文件存储、音视频通话等核心技术
- 适合作为软件工程课程设计的综合实践项目

## 二、功能规划与验收

### Q4：你是如何进行功能验收的？验收标准是什么？

**答：** 我制定了一套三层验收体系：

**第一层：单元验收（开发自测）**
- 每个功能开发完成后，开发人员先自测
- 后端：使用 Knife4j Swagger 页面逐个接口调试
- 前端：浏览器 DevTools + Vue DevTools 验证数据流

**第二层：功能验收（我来执行）**
- 按照我编写的测试用例逐条验收
- 测试环境：Windows 11 + Chrome 浏览器 + 本地服务
- 验收通过标准：
  - 正常流程无报错
  - 异常流程有合理错误提示
  - UI 无错位、无乱码、无样式丢失
  - 响应时间 < 2 秒

**第三层：集成验收（全员参与）**
- 前后端同时启动，多个测试账号模拟真实聊天场景
- 验证跨模块功能联动（如：A 发消息 → B 收到推送 → C 查看群消息不受影响）
- 验证数据持久化（重启服务后历史数据不丢失）

**验收清单示例：**

| 用户注册 | 输入合法信息后注册成功并自动登录 | 手动测试 | ✅ |
| 重复注册 | 用已注册用户名再次注册，提示错误 | 手动测试 | ✅ |
| 私聊消息 | A 发消息后 B 实时收到，延迟 <500ms | 手动测试 | ✅ |
| 图片发送 | 上传 ≤5MB 图片后双方可见 | 手动测试 | ✅ |
| 断线重连 | 关闭 WiFi 10 秒后恢复，自动重连 | 手动测试 | ✅ |
| ... | ... | ... | ... |

### Q5：你在测试中发现了哪些问题？是怎么推动解决的？

**答：** 我在功能验收中发现了 3 个 Bug，都进行了从发现到修复的完整闭环管理：

**Bug #1：ChatItem 列表项布局错位（P1 — 高优先级）**
- 发现过程：验收未读徽标功能时，发现整个会话列表的布局全部错乱——聊天项高度异常、文字不对齐
- 定位协助：我描述了现象（"会话列表突然全部错位，之前是正常的"），前端开发（王炳晟）通过 git diff 回溯代码变更，发现是 CSS 替换时 `display:flex` → `display:inline-block` 误扩散到了多个父级选择器
- 修复验证：前端精准替换后，我重新验收了会话列表的 5 个场景（有未读/无未读/免打扰/置顶/长昵称截断），全部恢复正常

**Bug #2：Safari 浏览器毛玻璃效果不生效（P2 — 中优先级）**
- 发现过程：我在 Safari 浏览器上验收时，发现登录页的玻璃面板是纯灰色不透明的
- 原因分析：Safari 的 `backdrop-filter` 需要 `-webkit-` 前缀
- 修复验证：前端添加 webkit 前缀后，我在 Safari + Chrome + Edge 三个浏览器上验证通过

**Bug #3：验证码图片首次加载控制台报错（P3 — 低优先级）**
- 发现过程：打开登录页时浏览器控制台出现红色报错（图片 src 为空）
- 原因分析：`captchaImage` 初始值为空字符串，`<img :src="">` 触发了无效请求
- 修复验证：前端添加 `v-show="captchaImage"` 条件渲染后，控制台不再报错

**推动修复的流程：** Bug 发现 → 详细记录复现步骤 → 提交给对应开发人员 → 修复后回归验证 → 测试报告中标记"已修复"

### Q9：需求分析文档中你重点分析了哪些方面？

**答：** 需求分析文档聚焦四个核心维度：

**1. 功能需求矩阵（核心产出）：**
- 将课设 11 条基本需求展开为 20+ 子功能点
- 额外识别 10 项扩展需求（离线消息、已读未读、好友管理、断线重连等）
- 每条需求标注优先级（Must/Should/Could）、技术可行性、实现状态

**2. 数据流分析：**
```
用户操作 → 客户端 → HTTP/WebSocket → 服务端
                                        ├→ MySQL（持久化）
                                        ├→ Redis（缓存/队列/在线状态）
                                        └→ MinIO（文件存储）
服务端 → WebSocket/HTTP Response → 客户端 → 界面更新
```

**3. 非功能需求：**
| 类别 | 指标 | 目标值 |
|------|------|--------|
| 性能 | 消息推送延迟 | < 500ms（局域网） |
| 可靠性 | 消息不丢失 | 先持久化再推送 |
| 安全性 | 密码存储 | BCrypt 加密 |
| 可用性 | 断线重连 | 自动重连 + 离线消息 |
| 可扩展性 | chat-server 水平扩展 | 支持多节点通过 Redis 路由 |

**4. 风险识别：**
| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| WebSocket 推送不稳定 | 中 | 高 | 心跳 + 自动重连 + 数据库持久化兜底 |
| MinIO Docker 容器异常 | 低 | 中 | 数据挂载宿主机目录 |
| 前端兼容性 | 低 | 低 | 优先支持 Chrome |

### Q12：如果重新规划这个项目，你会在流程上做什么调整？

**答：**

2. **UI 原型先行**：先出低保真原型（Figma/Sketch），确定 UI 风格后再开发前端，避免后期大面积返工
3. **接口 Mock**：后端未完成时前端用 Mock 数据先行开发，减少等待时间
4. **更规范的测试用例**：引入测试管理工具（如 TestLink），用例与需求建立追溯关系
5. **代码评审**：引入 Pull Request + Code Review 流程，提高代码质量
6. **自动化测试**：后端引入 JUnit + Testcontainers，前端引入 Jest，减少手动测试工作量

### Q14：三个角色是如何协作的？有没有出现过冲突？
