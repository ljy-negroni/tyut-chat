## 一、前端架构

### Q1：请介绍一下前端项目的整体架构与技术选型。

**答：** 我负责的前端项目 `chat-web` 是一个独立的 Vue 2.7 单页面应用（SPA）。

| UI 库 | Element UI | 2.15.14 | 表单、对话框、标签等基础组件 |
| 状态管理 | Pinia | 2.1.7 | 全局状态（用户、会话、好友、群组） |
| 路由 | Vue Router | 3.6.5 | SPA 页面切换 |

### Q2：为什么选择 Pinia 而不是 Vuex？有哪些 Store？各自管理什么状态？

**答：** 选择 Pinia 的原因：
1. **Vue 2.7 兼容**：Pinia 2.1 同时支持 Vue 2 和 Vue 3，为未来升级留空间
2. **TypeScript 友好**：虽然项目用 JS，但 Pinia 的类型推断比 Vuex 好
3. **模块化自然**：无需 nested modules，每个 Store 独立文件
4. **DevTools 支持**：Vue DevTools 可实时查看各 Store 状态

五个 Store 各有职责：

| Store | 核心状态 | 关键方法 |
|-------|---------|---------|
| userStore | userInfo（当前用户信息）、token | setUserInfo、clear |
| chatStore | conversations（会话列表）、activeConversation、unreadCount | setActive、insertMessage、readedMessage、remove |
| friendStore | friends（好友列表，按拼音分组） | findFriend、setDnd、addFriend、removeFriend |
| groupStore | groups（群组列表） | isGroup、setDnd、addGroup |
| configStore | fullScreen（全屏模式） | setFullScreen |

数据流向：
```
用户操作 → 组件 emit/click
         → Store action（异步 API 调用）
         → 更新 Store state（响应式）
         → 组件自动重新渲染
```

### Q3：axios 是怎么封装的？拦截器做了哪些处理？

**答：** 我在 `api/httpRequest.js` 中封装了 axios 实例：

```javascript
const http = axios.create({
    baseURL: process.env.VUE_APP_BASE_API,  // '/api'（开发环境代理到 8088）
    timeout: 30000,
    withCredentials: true
});
```

**请求拦截器：**
- 自动从 sessionStorage 读取 accessToken，添加到请求头 `accessToken`
- Token 不存在时不做处理（兼容白名单接口：/login /register /captcha）

**响应拦截器（核心逻辑）：**
```
response.data.code == 200  → 返回 response.data.data（业务数据）
response.data.code == 400  → 跳转根路径 /（Token 无效）
response.data.code == 401  → Token 过期，自动用 refreshToken 续期：
    1. 调用 PUT /refreshToken
    2. 保存新 Token 到 sessionStorage
    3. 用新 Token 重放原请求
    4. 用户无感知
其他错误码 → Message({type: "error"}) 弹出错误提示
```

**网络错误处理：**
- 400 → 参数错误提示
- 401 → 跳转登录页
- 404/500 → "服务器出了点小差"
- 501 → "服务器不支持当前请求"

这个封装让所有组件调用接口时只需关注业务数据，Token 管理、错误处理、自动续期全部由拦截器统一处理。

### Q4：前端如何配置开发环境代理？.env 文件的作用是什么？

**答：** 我在项目根目录配置了两个环境文件：

`.env.development`（开发环境）：
```
ENV = 'development'
VUE_APP_BASE_API = '/api'
VUE_APP_WS_URL = 'ws://localhost:8878/im'
VUE_APP_NAME = "TYUT CHAT"
```

`.env.production`（生产环境）：
```
ENV = 'production'
VUE_APP_BASE_API = 'https://your-domain.com/api'
VUE_APP_WS_URL = 'wss://your-domain.com/im'
VUE_APP_NAME = "TYUT CHAT"
```

`vue.config.js` 中配置开发代理：
```javascript
devServer: {
    proxy: {
        '/api': {
            target: 'http://127.0.0.1:8088',  // 转发到后端 platform
            changeOrigin: true,
            pathRewrite: { '^/api': '' }        // /api/login → /login
        }
    }
}
```

这样前端请求 `/api/login` → 代理转发到 `http://127.0.0.1:8088/login`，解决了开发时的跨域问题。

## 二、登录/注册页面

### Q5：登录/注册页的 UI 设计理念是什么？为什么采用这个风格？

**答：** 我设计的登录/注册页采用**macOS Vibrancy 暗色毛玻璃风格**，这是从 Apple 桌面应用设计语言中提炼的 Web 还原。

**设计核心理念：**

**1. 层级即色彩**
- 不使用渐变和发光来区分层级
- 纯靠背景深浅：`#0a0a0c`（背景画布）→ `rgba(28,28,30,0.72)`（玻璃面板）→ `rgba(28,28,30,0.6)`（输入框）
- 这是 macOS 的一贯做法：NSVisualEffectView 通过半透明 + 模糊制造深度感

**2. 左品牌右表单布局**
- 左侧展示"TYUT CHAT"衬线字体标题（Georgia / Noto Serif SC），建立品牌识别
- 右侧放置毛玻璃表单面板，保持功能区域紧凑专注
- 响应式 768px 断点：移动端自动切换为上下排列

**3. Three.js 3D 粒子背景**
- 12 个多面体（Icosahedron/Octahedron/Tetrahedron/TorusKnot）随机浮动
- 800+ 粒子构成 4 层光环
- 5 层 Shader 极光丝带
- 3 层 Torus 光环
- 全部使用极淡灰白色调，在纯黑背景上形成深邃的空间感
- 鼠标微移时相机跟随（lerp 平滑插值），增加交互感

**4. 1px 哲学**
- 所有分隔都是 1px 半透明边框 `rgba(255,255,255,0.08)`
- 不用 `box-shadow` 制造深度，不用 `border-2/border-4`
- 输入框 focus 时的反馈也只是边框变亮（0.1 → 0.22），无发光效果

**5. 无装饰主义**
- 按钮纯色 `#3a3a3c`，hover 微亮 `#4a4a4c`
- 没有渐变、没有发光、没有动画（仅有进入页面的淡入效果）
- 内容本身就是装饰

### Q6：登录/注册页的 Three.js 3D 场景是如何实现的？

**答：** Three.js 场景包含以下元素，全部用原生 WebGL 编写：

**1. 场景搭建：**
- `WebGLRenderer` + `antialias: true` + `alpha: true`（透明通道）
- 场景背景色 `#0a0a0c`，雾效同步
- 透视相机 44° FOV，位置 (0, 2, 26)，看向原点

**2. 浮动几何体（12 个）：**
```javascript
const geoSet = [
    new IcosahedronGeometry(0.1, 1),    // 二十面体
    new OctahedronGeometry(0.08, 0),    // 八面体
    new TetrahedronGeometry(0.09, 0),   // 四面体
    new TorusKnotGeometry(0.07, 0.025, 36, 6)  // 环结
];
// 每个几何体赋予 MeshStandardMaterial（暗灰金属质感）
// 随机分布在半径 7+ 的圆形轨道上
// 动画：X/Z 坐标按正弦/余弦轨迹浮动，Y 轴微动，自身缓慢旋转
```

**3. 粒子环（4 层，800+ 粒子）：**
- 使用 BufferGeometry + PointsMaterial + AdditiveBlending
- 粒子颜色采用灰白 HSL（饱和 0.04，亮度 0.35-0.6）
- 透明度 0.18，叠加混合制造光晕感
- 每秒绕 Y 轴微转

**4. Shader 极光丝带（5 层）：**
- 自定义 vertex shader + fragment shader
- 使用 sin/cos 组合波函数生成流动波纹
- `smoothstep` 制造渐变衰减
- 颜色从亮紫改为灰白五色：`['#3a3a44','#4a4a55','#353544','#454555','#3d3d48']`

**5. 性能优化：**
- `requestAnimationFrame` 驱动渲染循环
- 组件销毁时 `beforeDestroy` 中释放 WebGL 资源（`renderer.dispose()`）
- `PixelRatio` 限制为 `Math.min(devicePixelRatio, 2)`
- 使用 `Clock.getElapsedTime()` 获取时间差，动画帧率无关

### Q7：登录状态是如何持久化的？"自动登录"功能怎么实现？

**答：** 我设计了分层存储策略：

**Token 存储层级：**
| 存储位置 | 内容 | 生命周期 |
|----------|------|----------|
| sessionStorage | accessToken、refreshToken | 浏览器标签页关闭即清除 |
| localStorage | refreshToken、isAutoLogin、username | 持久化，除非手动清除 |

**自动登录流程：**
1. 用户登录时勾选"自动登录"→ `saveLoginSession()` 将 refreshToken 同时存入 localStorage
2. 下次打开页面 → `mounted()` 调用 `tryAuto()`：
   - 检查 `localStorage.getItem('isAutoLogin')` 是否为 true
   - 用 refreshToken 调用 `PUT /refreshToken` 获取新 accessToken
   - 成功后自动跳转聊天页，用户无需手动输入密码
3. 取消自动登录 → 清除 localStorage 中的 refreshToken

**安全考量：**
- accessToken 仅在 sessionStorage 中，浏览器关闭即失效
- refreshToken 虽然持久化，但仅用于续期接口，暴露面小
- 退出登录时清除所有 Token 存储

## 三、聊天界面

### Q8：聊天页面的组件结构是怎样的？会话列表和消息区如何通信？

**答：** 聊天页面（Chat.vue）的组件树：

```
Chat.vue
├── ResizableAside（可拖拽宽度的侧边栏，默认 260px）
│   ├── el-input（搜索框，过滤会话）
│   └── VirtualScroller（虚拟滚动列表）
│       └── ChatItem × N（每个会话条目）
│           └── HeadImage（头像组件）
└── ChatBox（消息区，v-if="activeConv"）
    ├── el-header（会话标题 + 群聊信息按钮）
    ├── el-main（消息列表，滚动加载历史）
    │   └── ChatMessageItem × N（每条消息气泡）
    │       └── HeadImage（头像）
    └── el-footer（输入工具栏 + 发送按钮）
        ├── 工具栏（表情、图片、文件、语音、通话）
        ├── ChatInput（文本输入框）
        └── el-button（发送按钮）
```

**会话列表 → 消息区的通信：**
- 通过 Pinia `chatStore` 共享状态
- 点击 ChatItem → `chatStore.setActive(conv.key)` → ChatBox 响应式获取 `chatStore.activeConversation` → 拉取该会话的历史消息

**消息列表性能优化：**
- `VirtualScroller` 虚拟滚动：只渲染可视区域的 ChatItem，列表上千条会话也不会卡
- 消息列表使用 `v-for` + `:key="m.localId"` 确保最小化 DOM 更新
- 新消息到达时只追加一条 ChatMessageItem，不重新渲染整个列表
- 上滑加载历史消息：监听 scroll 事件，触顶时请求上一页

### Q9：消息气泡的样式是怎么设计的？如何区分不同类型的消息？

**答：** 消息气泡通过 `ChatMessageItem` 组件的 `message.type` 字段区分渲染：

| type | 含义 | 渲染方式 |
|------|------|----------|
| 0 | 文字 | `<span class="message-text" v-html="htmlText">`（支持 URL 自动链接） |
| 1 | 图片 | `<img :src="thumbUrl" @click="查看原图">`（懒加载） |
| 2 | 文件 | 文件卡片（文件名 + 文件大小 + 下载图标） |
| 3 | 语音 | `<audio controls>` 播放器 |
| 4 | 视频 | 视频播放器 |
| 10 | 撤回 | "你撤回了一条消息" / "对方撤回了一条消息" |
| 20/21 | 时间/系统提示 | 灰色居中文字 |
| 40/41 | 音视频通话 | 通话类型图标 + "通话时长" |

**气泡样式（macOS 风格）：**
- 对方消息：灰色底 `#e9e9eb`（iMessage 经典色），黑色文字，圆角 `8px 18px 18px 18px`，靠左显示，左侧带头像
- 自己消息：蓝色底 `#0a84ff`（macOS 标准蓝），白色文字，圆角 `18px 8px 18px 18px`，靠右显示，右侧带头像
- 消息状态：发送中显示 loading 动画，发送失败显示红色感叹号（可点击重发），已读显示"已读"标记

### Q10：表情功能是怎么实现的？emoji 如何渲染？

**答：** 表情功能分两层实现：

**1. emoji 表情（Unicode 原生）：**
- 用户在输入框中直接输入 emoji（系统输入法），以 Unicode 字符存储
- 聊天界面中使用系统默认 emoji 字体渲染，无需额外资源
- 大尺寸 emoji（单独发送一个 emoji 时）自动放大字号

**2. 自定义表情包（GIF 图片）：**
- 表情图片存放在 `src/assets/emoji/` 目录下
- `api/emotion.js` 提供表情映射表（表情名称 → 图片路径）
- 输入框工具栏点击表情图标 → 弹出表情选择面板
- 选择表情后插入 `[表情名]` 文本标记
- 消息渲染时 `api/emotion.js` 的 `transform()` 方法将 `[表情名]` 替换为 `<img src="emoji/xxx.gif">`

**3. 表情渲染流程：**
```javascript
// 发送时：文本中包含 [微笑]
"你好[微笑]"

// 渲染时：$emo.transform() 替换
"你好<img src='/emoji/weixiao.gif' class='emoji-normal'>"
```

## 四、文件与图片

### Q11：图片上传与预览是怎么实现的？如何处理大图？

**答：** 图片上传采用分步处理：

**上传前端：**
1. 用户点击"发送图片"→ 触发隐藏的 `<input type="file" accept="image/*">`
2. `FileUpload` 组件校验：仅允许 JPEG/PNG/WebP/GIF，最大 5MB
3. `POST /image/upload`（multipart/form-data），显示上传进度

**图片压缩：**
- 前端不做压缩，由后端 Thumbnailator 统一处理
- 后端生成 300px 宽的缩略图，保留原图
- 消息体只存 URL，不存 base64（减少消息体大小）

**图片展示：**
- 聊天列表显示缩略图（`thumbUrl`），减少带宽消耗
- 缩略图样式使用 `object-fit: cover`，宽高自适应（最长边 ≤360px，最短边 ≥60px，保持原始比例）
- 点击缩略图 → 弹出全屏遮罩层 → 显示原图（`originUrl`）
- 全屏图片支持鼠标滚轮缩放、拖拽平移

### Q12：文件上传下载是怎么实现的？有没有断点续传？

**答：** 文件上传流程：

1. 用户点击"发送文件"→ 选择文件
2. FileUpload 校验：最大 10MB，无类型限制
3. `POST /file/upload`（multipart/form-data）
4. 后端上传至 MinIO → 返回文件 URL + 文件名 + 文件大小
5. 前端渲染为文件卡片：
   - 文件图标（el-icon-document）
   - 文件名（可点击下载）
   - 文件大小（自动转换 B/KB/MB）
6. 接收方点击文件名 → 浏览器直接打开 URL 下载

**下载实现：**
- 文件 URL 是 MinIO 的直接访问地址
- 前端使用 `<a href="url" download="filename">` 触发浏览器下载
- 或使用 `el-link` 组件渲染可点击链接

**断点续传**：目前尚未实现，属于课设扩展需求中的规划项。技术方向上可以使用 `Blob.slice()` 分片上传 + Redis 记录上传进度 + 服务端合并分片。

## 五、UI 设计系统

### Q13：你是如何保证整个前端 UI 风格统一的？

**答：** 我建立了一套非正式的 macOS Vibrancy 暗色设计规范，所有新增组件遵循：

**色彩令牌：**
```
背景层级（由深到浅）：
  #0a0a0c  — 页面背景（最深）
  #1c1c1e  — 侧边栏/面板底色
  #2c2c2e  — 卡片/消息气泡底色
  #3a3a3c  — 按钮/选中态

边框：
  1px rgba(255,255,255,0.08) — 面板边框
  1px rgba(255,255,255,0.10) — 输入框边框
  1px rgba(255,255,255,0.22) — 输入框 focus

文字：
  rgba(255,255,255,0.92) — 标题/重点文字
  rgba(255,255,255,0.70) — 正文
  rgba(255,255,255,0.35) — 辅助文字/placeholder
  rgba(255,255,255,0.25) — 极淡文字
```

**圆角规范：**
- 大面板：16px
- 输入框/按钮：10px
- 消息气泡：18px（不对称圆角模拟 iMessage）
- 头像/徽标：50%（圆形）/ 9px（半圆形徽标）

**禁止项：**
- 渐变背景（`linear-gradient`、`radial-gradient`）
- 大阴影（`box-shadow` 超过 2px 扩散）
- 装饰动画（`animation: pulse/bounce/spin`）
- 圆角超过 18px（不使用 `border-radius: 9999px`）
- 亮色作为大面积背景

**统一检查：** 每次新增或修改组件后，我会切换聊天/好友/群聊三个标签页，确认搜索框、列表项、在线状态、文本颜色风格一致。

### Q14：响应式设计是怎么考虑的？移动端能用吗？

**答：** 当前版本主要面向桌面端（1280×720 以上分辨率），但做了基础的响应式适配：

**登录/注册页：**
- `@media (max-width: 768px)` 断点
- 桌面端：左右分栏（品牌 + 表单）
- 移动端：上下堆叠（品牌区缩小 + 表单全宽）

**侧边栏：**
- `ResizableAside` 组件支持拖拽调整宽度（200px - 500px）
- 宽度持久化到 localStorage，下次打开恢复

**聊天页：**
- 消息气泡 `max-width: 80%`（桌面）/ `90%`（移动）
- 图片缩略图 `maxSize: 360px`（全屏模式）/ `240px`（普通）

**移动端的限制：**
- WebRTC 音视频通话在移动浏览器兼容性有限
- Three.js 粒子场景在低性能手机上可能掉帧
- 建议后续开发 React Native / Flutter 移动客户端

## 六、前后端联调

### Q15：前后端是如何协作联调的？接口变更怎么同步？

**答：** 联调流程：

1. **接口先行**：后端（李俊毅）先定义 REST API 接口（URL、Method、Request Body、Response），提供 Swagger 文档（`http://localhost:8088/doc.html`）
2. **协议约定**：WebSocket 消息帧格式约定（type、sendId、recvId、content 的 JSON 结构）
3. **前端对接**：我根据 Swagger 文档编写 API 调用代码，WebSocket 帧格式编写解析函数
4. **联调测试**：前后端同时启动，通过浏览器 DevTools Network/WS 标签查看实际请求与响应
5. **问题排查**：
   - HTTP 问题：看 Network 标签的 Request/Response
   - WebSocket 问题：看 WS 标签的 Messages 帧序列
   - 跨域问题：验证 vue.config.js 代理配置
   - 数据格式问题：对照 Swagger 定义的字段名

**接口变更同步：**
- 后端修改接口后同步更新 Swagger 文档
- 前端报错时首先对照 Swagger 检查字段名、数据类型是否匹配
- 使用 Git 分支管理，接口变更在单独分支完成后再合并

### Q16：联调中遇到最棘手的问题是什么？怎么解决的？

**答：** 最棘手的问题是**验证码图片加载失败**，根源有三个层面：

**第一层（后端）：** `/captcha` 接口返回 401
- 原因：AuthInterceptor 默认拦截所有请求，未登录状态下无法调用 /captcha
- 解决：在 MvcConfig 的 excludePathPatterns 中添加 `/captcha`

**第二层（前端）：** 图片 URL 格式错误 `ERR_INVALID_URL`
- 原因：后端 Hutool `getImageBase64Data()` 返回值已包含 `data:image/png;base64,` 前缀，前端又拼接了一次
- 解决：去掉前端拼接，直接使用 `d.captchaImage` 作为图片 src

**第三层（前端）：** 图片加载失败无提示
- 原因：`getCaptcha()` 方法的 Promise 没有 `.catch()` 错误处理
- 解决：添加 `.catch(e => console.error('获取验证码失败', e))`

这个问题的排查让我体会到：前后端联调时，错误可能发生在任意一层，需要逐层定位。浏览器 DevTools 的 Network 标签是最有效的排查工具。
