## 2026.06.15（需求分析与架构设计）

### 当日工作计划
- 完成项目需求分析文档编写
- 完成系统架构设计与技术选型
- 确定模块划分与依赖关系
- 搭建基础开发环境

### 当日工作内容

**1. 需求分析（2 小时）**
- 分析课设要求，明确项目目标：构建面向太原理工大学师生的校园即时通讯 Web 应用
- 梳理核心功能需求：用户注册登录、私聊消息、群聊消息、图片/文件/语音发送、音视频通话（WebRTC）、好友管理、群组管理、消息撤回删除、敏感词过滤
- 撰写《需求分析文档》，包含功能需求矩阵、非功能需求（性能、安全、可扩展性）、数据流图

**2. 系统架构设计（2 小时）**
- 确定采用 B/S 架构 + 微服务思想，后端拆分为 4 个 Maven 模块：
  - `chat-common`：公共包，定义共享枚举、工具类、JWT 工具
  - `chat-client`：消息推送 SDK，封装 Redis 队列写入逻辑，供 platform 调用
  - `chat-server`：WebSocket 推送服务，基于 Netty，消费 Redis 队列向客户端推送消息
  - `chat-platform`：业务平台服务，基于 Spring Boot 3.3，提供 REST API
- 前端独立工程 `chat-web`，Vue 2 + Element UI + Pinia 状态管理
- 依赖方向严格单向：common ← client ← platform，common ← server

**3. 技术选型（1 小时）**
- 后端：Spring Boot 3.3、Netty 4.x（WebSocket 长连接）、MyBatis-Plus 3.5（ORM）、Druid（连接池）、Redis + Redisson（缓存/分布式锁/消息队列）、MinIO（对象存储）、JWT（无状态认证）、Knife4j（API 文档）
- 前端：Vue 2.7、Element UI 2.15、Pinia 2.1（状态管理）、Axios（HTTP 请求）、Three.js 0.160（3D 背景）、SCSS（样式预处理）
- 存储：MySQL 8.0（关系型数据）、Redis 6.2（缓存/消息队列）、MinIO（图片/文件/语音）
- 构建：Maven 多模块 + JDK 17

**4. 开发环境搭建（2 小时）**
- 安装配置 JDK 17，设置 JAVA_HOME 环境变量
- 安装 Maven 3.9，配置 settings.xml 镜像源为阿里云
- 安装 MySQL 8.0，创建数据库 `im_platform_open`，设置字符集 utf8mb4
- 安装 Docker Desktop（Windows 11 Home，WSL 2 引擎），拉取并启动 Redis 6.2 容器（端口 6379）
- 拉取并启动 MinIO 容器（端口 9000/9001），数据目录挂载到 E 盘
- 初始化 Git 仓库，配置 .gitignore 忽略 node_modules、target、*.iml 等

### 遇到的问题及解决方法
| 问题 | 原因 | 解决方法 |
|------|------|----------|
| Redis 启动后无法连接 | Docker 容器未绑定端口 | `-p 6379:6379` 映射端口 |
| MinIO Access Key 认证失败 | 配置文件中密钥与容器不一致 | 统一使用 admin/admin123456 |
| Windows Home 版不支持 Hyper-V | 系统版本限制 | 使用 WSL 2 引擎替代 |

## 2026.06.17（消息系统与 WebSocket 推送）

### 当日工作计划
- 完成 WebSocket 推送服务（chat-server）
- 实现私聊消息发送与接收流程
- 实现消息已读/未读状态同步
- 实现 Redis 消息队列跨节点路由

### 当日工作内容

**1. Netty WebSocket 服务端（3 小时）**
- 编写 WebSocketServer 启动类，绑定端口 8878，配置 BossGroup/WorkerGroup 线程池
- 编写 IMChannelHandler，处理 channelActive（连接建立）、channelInactive（断开）、channelRead（消息读取）
- 编写 LoginProcessor，解析客户端登录帧，绑定 userId 与 Channel 映射关系，存入本地 ConcurrentHashMap
- 编写 PrivateMessageProcessor，处理私聊消息的转发逻辑：
  - 接收方在线 → 直接通过 WebSocket 推送
  - 接收方离线 → 不处理，等待对方上线后拉取历史

**2. 消息推送链路（2 小时）**
- 设计 Redis List 消息队列模型：每台 chat-server 实例持有独立的队列 key `im:unread:{serverId}`
- 用户登录时，在 Redis 中记录 `im:user_server:{userId}` → serverId 映射
- Platform 发送消息时：查询接收方所在 server → 将消息 JSON 写入对应 Redis List
- Server 消费线程：定时 BLPOP 自己的队列，取出消息后通过 WebSocket 推送给目标用户

**3. 私聊消息 API（2 小时）**
- 编写 PrivateMessageController，提供：
  - POST /message/private/send：发送私聊消息
  - GET /message/private/history：拉取历史消息（分页，按时间倒序）
  - DELETE /message/private/delete：删除单条消息
  - PUT /message/private/recall：撤回消息（2 分钟内）
- 编写 PrivateMessageService 实现，处理消息体序列化、敏感词过滤、状态流转（SENDING → DELIVERED → READED）

**4. 消息状态管理（1.5 小时）**
- 定义 MESSAGE_STATUS 枚举：FAILED(-2)、SENDING(-1)、PENDING(0)、DELIVERED(1)、RECALL(2)、READED(3)
- 实现已读回执：客户端收到消息后发送 ACK 帧，服务端更新消息状态为 DELIVERED
- 实现已读上报：用户进入会话时发送 READ 帧，服务端批量更新该会话所有消息为 READED

### 遇到的问题及解决方法
| 问题 | 原因 | 解决方法 |
|------|------|----------|
| WebSocket 连接频繁断开 | Netty 未配置心跳 | 添加 IdleStateHandler，60s 读写空闲发送 PING |
| 消息乱序问题 | 异步推送未排队 | 使用 LinkedBlockingQueue 顺序处理同一用户的推送 |
| 大消息体导致 OOM | 图片 base64 直接存消息体 | 图片改为上传 MinIO，消息体只存 URL |

## 2026.06.22（验证码系统、安全加固、环境配置本地化）

### 当日工作计划
- 实现图形验证码功能，防止机器人注册
- 将数据库密码等敏感配置改为环境变量占位
- 解决 MinIO 生产环境配置问题

### 当日工作内容

**1. 图形验证码系统（3 小时）**
- 在 chat-platform pom.xml 中添加 Hutool 5.8.28 依赖（使用 CaptchaUtil 工具类）
- 编写 CaptchaController 控制器：
  - GET /captcha：调用 `CaptchaUtil.createLineCaptcha(120, 42, 4, 20)` 生成 4 位数字线段干扰验证码
  - 生成 UUID 作为 key，将验证码文本存入 Redis：`SET captcha:{uuid} code EX 300`（5 分钟过期）
  - 返回 JSON：`{captchaKey: uuid, captchaImage: base64Data}`（图片 Base64 编码，直接嵌入 img 标签）
- 修改 RegisterDTO：新增 `captchaKey`（@NotEmpty）、`captchaCode`（@NotEmpty）字段
- 修改 LoginDTO：新增 `captchaKey`、`captchaCode` 字段（非必填，兼容自动登录场景）
- 修改 UserServiceImpl.register()：从 Redis 读取验证码文本，忽略大小写比对，校验后删除 key 防止复用
- 修改 UserServiceImpl.login()：仅当 captchaKey 非空时校验验证码（注册后自动登录场景不过验证码）
- 修改 MvcConfig 拦截器白名单：添加 `/captcha` 路径，允许未登录时获取验证码
- 前端 Login.vue 与 Register.vue 各添加验证码输入框 + 图片展示区，图片点击可刷新

**2. 配置文件安全加固（1 小时）**
- 将 application-dev.yml 中的硬编码数据库连接信息改为环境变量占位：
  - `url: jdbc:mysql://${MYSQL_HOST:localhost}:${MYSQL_PORT:3306}/...`
  - `username: ${MYSQL_USER:root}`
  - `password: ${MYSQL_PASSWORD:root}`
- Redis 连接同理：`host: ${REDIS_HOST:localhost}`、`port: ${REDIS_PORT:6379}`
- MinIO 外网访问 domain 改为：`http://${MINIO_DOMAIN:localhost}:9000`

**3. MinIO 配置修复与测试（1.5 小时）**
- 发现 FileServiceImpl 初始化逻辑缺陷：`setBucketPublic()` 仅在 bucket 首次创建时调用，若 bucket 已存在则跳过，导致重启后 bucket 权限丢失
- 修改逻辑：将 `setBucketPublic()` 移到 `if (!bucketExists)` 外部，确保每次启动都调用
- 测试：上传图片 → 浏览器直接访问 URL → 确认 200 响应
- 发现 MinIO domain 写死为旧 IP 192.168.2.67，改为环境变量后当前 IP 为 192.168.31.52（WiFi DHCP 变化）

### 遇到的问题及解决方法
| 问题 | 原因 | 解决方法 |
|------|------|----------|
| 验证码图片不显示（前端） | AuthInterceptor 拦截了未登录的 /captcha 请求 | MvcConfig 添加 /captcha 白名单 |
| 验证码图片 URL 错误 ERR_INVALID_URL | Hutool getImageBase64Data() 已含 `data:image/png;base64,` 前缀，前端又拼接了一次 | 前端去掉前缀拼接，直接使用返回值 |
| 图片外网访问超时 | MinIO domain 指向旧 IP | 改为环境变量 ${MINIO_DOMAIN}，默认 localhost |
| 平台重启后图片 403 | setBucketPublic 只在创建时调用 | 移到 if 外，每次启动都执行 |

## 技术总结

### 项目架构

```
tyut-chat/
├── chat-common/          # 公共包：枚举、工具类
├── chat-client/           # 推送 SDK：Redis 队列写入
├── chat-server/           # WebSocket 推送服务（Netty，端口 8878）
├── chat-platform/         # 业务平台服务（Spring Boot，端口 8088）
│   ├── controller/        # REST 控制器（Login、File、Friend、Group、Message、Captcha 等）
│   ├── service/           # 业务逻辑层
│   ├── mapper/            # MyBatis-Plus 数据访问层
│   ├── entity/            # 数据库实体
│   ├── dto/               # 数据传输对象
│   ├── vo/                # 视图对象
│   ├── config/            # Spring 配置（Redis、MinIO、JWT、MVC、Swagger）
│   ├── interceptor/       # 认证拦截器、XSS 拦截器
│   └── thirdparty/        # 第三方服务封装（MinIO）
└── chat-web/              # Vue 2 前端
    ├── src/view/          # 页面组件（Login、Register、Chat、Friend、Group、Home）
    ├── src/components/    # 通用组件（HeadImage、ChatItem、ChatMessageItem、FriendItem 等）
    ├── src/api/           # API 层（axios 封装、WebSocket 客户端、auth 工具）
    ├── src/store/         # Pinia 状态管理
    └── src/assets/        # 静态资源（iconfont、emoji、style）
```

### 核心技术亮点
1. **跨节点消息推送**：基于 Redis List 实现多 chat-server 实例的消息路由，每个节点独立消费自己的队列
2. **无状态认证**：JWT 双 token 机制（accessToken 短期 + refreshToken 长期），支持自动续期
3. **图形验证码**：Hutool 线段干扰 + Redis 5 分钟过期 + 一次性使用
4. **对象存储**：MinIO 自建 OSS，bucket 公开读策略，图片自动缩略图
5. **3D 粒子背景**：Three.js 原生 WebGL，多面体浮动 + 粒子环 + Shader 极光丝带，性能优化（requestAnimationFrame + lerp 平滑）
6. **macOS Vibrancy 设计系统**：暗色层级（#0a0a0c → #1c1c1e → #2c2c2e → #3a3a3c）、系统级 backdrop-blur、1px 半透明边框、无渐变无大阴影
