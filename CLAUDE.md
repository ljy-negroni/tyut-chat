# tyut-chat 项目规约

> 本文件是本项目的**强制规约，永久生效**。
> **任何 AI 助手在本项目开工前——含会话启动、上下文重置、接手新任务——必须先完整读本文件，再动手。** 与规约冲突的“方便做法”一律禁止。

## 一、开工先读
每次上下文用完 / 会话开始，第一件事：读本文件 + 当前计划书，确认任务边界与待办。宁可多问，不得擅破规约。

## 二、架构与模块边界（不许偏移）
后端四模块 + 一前端，职责固定，禁止拆并、越界、反向依赖：
- `chat-common` 公共包：所有后端可依赖，不依赖任何兄弟模块。
- `chat-client` 推送 SDK：仅依赖 chat-common。
- `chat-server` WebSocket 推送服务：仅依赖 chat-common。
- `chat-platform` 业务平台服务（HTTP）：依赖 chat-client。
- `chat-web` Vue2 前端：独立工程。

依赖方向严格单向。Java 包名 `com.tyut.im*` 保持不变。

## 三、技术栈（锁定，禁止更改）
后端 Spring Boot 3.3 + Netty + MyBatis-Plus + Druid + Redis + Redisson + MinIO + JWT + Knife4j；前端 Vue 2 + Element UI + Pinia；存储 MySQL 8.0 + Redis + MinIO；JDK 17 + Maven 多模块。未经书面同意不得引入新框架、替换技术栈或大版本升级。

## 四、Git 提交（小而精）
- 一个 commit 只做一件事，**严禁把多个任务揉在一起提交**。
- 提交前必须本地编译 / 测试通过，红代码不进仓库。
- commit message 用中文祈使句，简短点题，必要时附正文说明动机。
- 禁止 AI 署名（见第八条）。

## 五、正确性保障
所有改动须经编译与（如有）测试验证，确认正确方可提交；涉及配置 / 依赖 / SQL 的改动须额外自查副作用。

## 六、方案决策（调研 + 人选）
存在多种实现路径时，先调研，给出至少两个可选方案及取舍，**由人选择**，不擅自拍板；默认不引入新依赖，确需引入须说明理由并待人确认。

## 七、大规模任务与计划书
- 大规模 / 跨模块任务须先写**任务计划书**（本仓库内，如 `PLAN.md`），列目标、拆解步骤、影响范围、风险，**经人审查通过后才能动工**。
- 所有改动与新增必须先登记进计划书，按计划书安排推进；临时改动事后补登。
- 计划书优先于即兴编码。

## 八、署名禁令（永久生效）
commit message、PR、代码、文档中**一律不得署名 Claude 或克劳德**，禁止添加 `Co-Authored-By: Claude` 等任何 AI 协作者标识。本项目归作者本人所有，不挂 AI 协作者。

## 九、文档管理
- 所有详细文档统一存放 `docs/`，按类别分子目录（设计 `docs/design/`、接口 `docs/api/`、部署 `docs/deploy/`、计划书 `docs/plan/` 等）。
- README 只放项目概览与快速上手，深入内容进 `docs/`。
- 文档用中文 Markdown，命名清晰达意。

## 十、敏感信息管理
- 数据库密码、JWT secret、MinIO 密钥、第三方密钥等不得硬编码或提交进仓库。
- 配置中敏感项用环境变量 `${VAR}` 占位，或放进 .gitignore 忽略的本地配置文件覆盖。
- 禁止把真实凭证写进代码、注释或示例。

## 十一、数据库变更脚本化
- 所有表结构（DDL）与重要数据变更必须写成版本化 SQL，放进 `db/更新脚本/`，命名带版本号与简述（如 `v_4.1_新增好友备注.sql`）。
- 禁止直接连库手动改表不留痕；脚本须能复现库结构。

## 十二、依赖版本统一
- 新增第三方依赖先在根 pom `<dependencyManagement>` 统一声明版本，子模块引用不写 `<version>`。
- 禁止子模块各自声明同名依赖的不同版本；版本升级统一改根 pom。

## 十三、编码与换行符
- 源码与配置统一 UTF-8；文本文件换行符统一 LF，仓库根用 `.gitattributes` 强制（如 `* text=auto eol=lf`）。
- 代码注释与文档用中文，标识符用英文。
